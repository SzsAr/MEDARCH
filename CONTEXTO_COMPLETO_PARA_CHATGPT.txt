PARTE 1: DESCRIPCIÓN GENERAL DEL PROYECTO
═══════════════════════════════════════════════════════════════════════════════

NOMBRE: MEDARCH
TIPO: Sistema de gestión documental para historias clínicas digitalizadas
OBJETIVO: Digitalizar archivos físicos históricos (10-20 años) de forma 
          estructurada, consultable y auditable

PROPÓSITO PRINCIPAL:
- Capturar documentos escaneados desde carpeta local
- Clasificarlos manualmente (OCR es solo apoyo)
- Almacenarlos en NAS de forma estructurada
- Registrar metadatos en PostgreSQL
- Mantener trazabilidad completa con auditoría

PARTE 4: SCHEMA DE BASE DE DATOS
═══════════════════════════════════════════════════════════════════════════════

TABLA: gesdoc.usuarios
─────────────────────────────────────────────────────────────────────────────

CREATE TABLE usuarios (
  id_usuario SERIAL PRIMARY KEY,
  usuario VARCHAR(50) NOT NULL UNIQUE,
  nombre VARCHAR(150) NOT NULL,
  rol VARCHAR(50) NOT NULL CHECK (rol IN ('CONSULTA','ARCHIVO','SUPERADMIN')),
  activo BOOLEAN DEFAULT TRUE,
  fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  password_hash VARCHAR(255) NOT NULL
);

CAMPOS:
- id_usuario: Identificador único (auto-incremento)
- usuario: Nombre de usuario único (3-50 caracteres)
- nombre: Nombre completo (2-150 caracteres)
- rol: CONSULTA | ARCHIVO | SUPERADMIN
- activo: Boolean, permite soft-delete y toggle
- fecha_creacion: Timestamp automático
- password_hash: Hash bcrypt de la contraseña (nunca se retorna)

TABLA: gesdoc.pacientes
─────────────────────────────────────────────────────────────────────────────

CREATE TABLE pacientes (
  id_paciente SERIAL PRIMARY KEY,
  numero_documento VARCHAR(20) NOT NULL UNIQUE,
  nombre VARCHAR(150),
  fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

TABLA: gesdoc.tipos_documento
─────────────────────────────────────────────────────────────────────────────

CREATE TABLE tipos_documento (
  id_tipo SERIAL PRIMARY KEY,
  codigo VARCHAR(10) NOT NULL UNIQUE,
  descripcion VARCHAR(100) NOT NULL,
  activo BOOLEAN DEFAULT TRUE
);

Ejemplos: HC (Historia Clínica), LAB (Laboratorio), IMA (Imagen), RX (Radiografía)

TABLA: gesdoc.documentos (MODELO HÍBRIDO)
─────────────────────────────────────────────────────────────────────────────

CREATE TABLE documentos (
  id_documento BIGSERIAL PRIMARY KEY,
  
  -- DATOS INICIALES (WATCHER)
  ruta_temporal VARCHAR(500) NOT NULL,
  nombre_archivo_original VARCHAR(255) NOT NULL,
  fecha_carga TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  -- ESTADO
  estado VARCHAR(20) NOT NULL DEFAULT 'PENDIENTE'
    CHECK (estado IN ('PENDIENTE','EN_REVISION','PROCESADO','ERROR')),
  
  -- DATOS COMPLETADOS EN PROCESAMIENTO
  id_paciente INT,
  id_tipo INT,
  fecha DATE,
  consecutivo INT,
  
  nombre_archivo VARCHAR(255),
  ruta_archivo VARCHAR(500),
  fecha_procesado TIMESTAMP,
  
  -- RELACIONES
  CONSTRAINT fk_doc_paciente FOREIGN KEY (id_paciente)
    REFERENCES pacientes(id_paciente),
  
  CONSTRAINT fk_doc_tipo FOREIGN KEY (id_tipo)
    REFERENCES tipos_documento(id_tipo),
    
  -- EVITAR DUPLICADOS LÓGICOS
  UNIQUE (id_paciente, id_tipo, fecha, consecutivo) WHERE estado = 'PROCESADO'
);

CICLO DE VIDA DE DOCUMENTO:

PENDIENTE    → Nuevos, esperando revisión manual
  ↓
EN_REVISION  → Usuario revisa OCR y completa datos
  ↓
PROCESADO    → Validado, guardado en NAS, archivo local a PROCESADOS
  ↓ (error en proceso)
ERROR        → Problemas, archivo va a ERROR, logs en BD

TABLA: gesdoc.log_auditoria
─────────────────────────────────────────────────────────────────────────────

CREATE TABLE log_auditoria (
  id_log BIGSERIAL PRIMARY KEY,
  id_documento BIGINT,
  id_usuario INT NOT NULL,
  accion VARCHAR(50) NOT NULL,
  detalle VARCHAR(500),
  fecha TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  CONSTRAINT fk_log_doc FOREIGN KEY (id_documento)
    REFERENCES documentos(id_documento),
  
  CONSTRAINT fk_log_usuario FOREIGN KEY (id_usuario)
    REFERENCES usuarios(id_usuario)
);

ACCIONES: CREAR, ACTUALIZAR, PROCESAR, ERROR, VALIDAR_OCR, etc.

PARTE 5: REGLAS DE NEGOCIO
═══════════════════════════════════════════════════════════════════════════════

1. CICLO DE VIDA DEL DOCUMENTO
   ✓ Un PDF = un solo tipo de documento
   ✓ Un PDF = una sola atención de paciente
   ✓ Estados: PENDIENTE → EN_REVISION → PROCESADO (o ERROR)
   ✓ No se permite volver atrás de PROCESADO (inmutable)

2. VALIDACIÓN Y OCR
   ✓ OCR es SOLO ASISTIVO
   ✓ USUARIO SIEMPRE valida y completa datos manualmente
   ✓ Usuario puede rechazar sugerencias OCR
   ✓ Información final es responsabilidad del usuario

3. NOMBRES DE ARCHIVO
   ✓ Formato obligatorio: TIPO_CEDULA_FECHA_CONSECUTIVO.pdf
   ✓ Ejemplo: HC_24909345_2014-12-01_001.pdf
   ✓ No se permiten cambios después de guardado

4. DUPLICADOS LÓGICOS
   ✓ Prevención: UNIQUE (id_paciente, id_tipo, fecha, consecutivo)
   ✓ Solo para documentos PROCESADOS
   ✓ Aplicable a nivel de BD

5. AUDITORÍA
   ✓ TODA acción significativa se registra en log_auditoria
   ✓ Quién: id_usuario
   ✓ Qué: acción
   ✓ Cuándo: fecha
   ✓ Cuál documento: id_documento
   ✓ Detalles adicionales: detalle

6. ALMACENAMIENTO
   ✓ BD es VERDAD LÓGICA
   ✓ NAS es VERDAD FÍSICA
   ✓ Si hay conflicto, BD siempre gana
   ✓ Archivos locales se limpian después de 7 días

7. ROLES Y PERMISOS
   ✓ CONSULTA: Solo lectura de documentos
   ✓ ARCHIVO: CRUD de documentos, validación OCR
   ✓ SUPERADMIN: Gestión de usuarios, todo acceso

8. CONTRASEÑAS Y SEGURIDAD
   ✓ Mínimo 8 caracteres
   ✓ Hash con bcrypt
   ✓ NUNCA retornar password_hash
   ✓ JWT con expiración 1 hora

AUTENTICACIÓN:

POST /auth/login

  Autenticación: ❌ Pública

USUARIOS - CRUD:
───────────────────────────────────────────────────────────────────────────────

1. POST /users (CREAR USUARIO)
   Autenticación: ❌ Pública (permite crear primer SUPERADMIN)
   Entrada:
   {
     "usuario": "string (3-50 caracteres)",
     "nombre": "string (2-150 caracteres)",
     "password": "string (8-128 caracteres)",
     "rol": "CONSULTA | ARCHIVO | SUPERADMIN"
   }
   Salida (201 Created):
   {
     "id_usuario": int,
     "usuario": "string",
     "nombre": "string",
     "rol": "string",
     "activo": boolean,
     "fecha_creacion": "ISO timestamp"
   }
   Errores:
   - 400 Bad Request: Validación falla
   - 409 Conflict: Usuario ya existe
   - 500 Internal Server Error: Error BD

2. GET /users (LISTAR USUARIOS)
   Autenticación: ✅ JWT + SUPERADMIN
   Header: Authorization: Bearer <token>
   Salida (200 OK):
   [
     { "id_usuario": 1, "usuario": "...", "nombre": "...", ...},
     { "id_usuario": 2, "usuario": "...", "nombre": "...", ...}
   ]
   Errores:
   - 401 Unauthorized: Sin token o token inválido
   - 403 Forbidden: No es SUPERADMIN
   - 500 Internal Server Error: Error BD

3. GET /users/{id_usuario} (OBTENER USUARIO)
   Autenticación: ✅ JWT + SUPERADMIN
   Header: Authorization: Bearer <token>
   Parámetro: id_usuario (integer)
   Salida (200 OK):
   {
     "id_usuario": int,
     "usuario": "string",
     "nombre": "string",
     "rol": "string",
     "activo": boolean,
     "fecha_creacion": "ISO timestamp"
   }
   Errores:
   - 401 Unauthorized: Sin token
   - 403 Forbidden: No es SUPERADMIN
   - 404 Not Found: Usuario no existe
   - 500 Internal Server Error: Error BD

4. PUT /users/{id_usuario} (ACTUALIZAR USUARIO)
   Autenticación: ✅ JWT + SUPERADMIN
   Header: Authorization: Bearer <token>
   Parámetro: id_usuario (integer)
   Entrada:
   {
     "nombre": "string (2-150 caracteres)",
     "rol": "CONSULTA | ARCHIVO | SUPERADMIN",
     "activo": boolean
   }
   Salida (200 OK):
   {
     "id_usuario": int,
     "usuario": "string",
     "nombre": "string",
     "rol": "string",
     "activo": boolean,
     "fecha_creacion": "ISO timestamp"
   }
   Notas:
   - NO se puede cambiar campo "usuario"
   - NO se modifica contraseña (usar PATCH)
   Errores:
   - 400 Bad Request: Rol inválido o validación falla
   - 401 Unauthorized: Sin token
   - 403 Forbidden: No es SUPERADMIN
   - 404 Not Found: Usuario no existe
   - 500 Internal Server Error: Error BD

5. PATCH /users/{id_usuario}/password (CAMBIAR CONTRASEÑA)
   Autenticación: ✅ JWT + SUPERADMIN
   Header: Authorization: Bearer <token>
   Parámetro: id_usuario (integer)
   Entrada:
   {
     "nueva_password": "string (8-128 caracteres)"
   }
   Salida (200 OK):
   {
     "message": "Contraseña actualizada correctamente"
   }
   Errores:
   - 400 Bad Request: Contraseña < 8 caracteres
   - 401 Unauthorized: Sin token
   - 403 Forbidden: No es SUPERADMIN
   - 404 Not Found: Usuario no existe
   - 500 Internal Server Error: Error BD

6. DELETE /users/{id_usuario} (TOGGLE ESTADO - ACTIVAR/DESACTIVAR)
   Autenticación: ✅ JWT + SUPERADMIN
   Header: Authorization: Bearer <token>
   Parámetro: id_usuario (integer)
   Salida (200 OK):
   {
     "message": "Usuario desactivado correctamente"
     O
     "message": "Usuario activado correctamente"
   }
   Comportamiento:
   - Si activo=true → pone activo=false (desactiva)
   - Si activo=false → pone activo=true (reactiva)
   - Es BIDIRECCIONAL (toggle)
   Errores:
   - 401 Unauthorized: Sin token
   - 403 Forbidden: No es SUPERADMIN
   - 404 Not Found: Usuario no existe
   - 500 Internal Server Error: Error BD

PARTE 8: FLUJO DE TRABAJO ACTUAL
═══════════════════════════════════════════════════════════════════════════════

FASE 1: INGESTA DE DOCUMENTOS (No implementado)

1. Usuario escanea PDF → C:\Escaneos\PENDIENTES
2. Watcher detecta nuevo archivo
3. Watcher crea registro en gesdoc.documentos:
   - estado = 'PENDIENTE'
   - ruta_temporal = C:\Escaneos\PENDIENTES\archivo.pdf
   - nombre_archivo_original = archivo.pdf
   - fecha_carga = NOW()
4. Watcher ejecuta OCR sobre PDF
5. OCR extrae sugerencias: cédula, tipo, fecha

FASE 2: VALIDACIÓN MANUAL (No implementado)

6. Frontend muestra documento en estado PENDIENTE
7. Usuario revisa OCR (cédula, tipo, fecha sugeridos)
8. Usuario VALIDA O RECHAZA cada sugerencia
9. Usuario completa:
   - id_paciente (seleccionar o crear)
   - id_tipo (seleccionar)
   - fecha (validar)
   - consecutivo (auto o manual)

FASE 3: PROCESAMIENTO (No implementado)

10. Sistema valida no duplicado: UNIQUE (id_paciente, id_tipo, fecha, consecutivo)
11. Sistema genera nombre:
    TIPO_CEDULA_FECHA_CONSECUTIVO.pdf
    Ejemplo: HC_24909345_2014-12-01_001.pdf
12. Sistema mueve archivo a NAS:
    /NAS/MEDARCH/24909345/HC/HC_24909345_2014-12-01_001.pdf
13. Sistema actualiza gesdoc.documentos:
    - estado = 'PROCESADO'
    - nombre_archivo = generado
    - ruta_archivo = ruta_NAS
    - fecha_procesado = NOW()
    - id_paciente = validado
    - id_tipo = validado
    - fecha = validado
    - consecutivo = validado
14. Sistema mueve archivo local a C:\Escaneos\PROCESADOS
15. Sistema registra en log_auditoria

PARTE 9: SEGURIDAD Y VALIDACIÓN
═══════════════════════════════════════════════════════════════════════════════

AUTENTICACIÓN:

✅ JWT (JSON Web Tokens)
  - Generado en login
  - Expira en 1 hora (configurable)
  - Contiene: id_usuario, usuario, rol
  - Algoritmo: HS256
  - Clave secreta: FROM .env (SECRET_KEY)

✅ Validación de Token:
  - Cada request protegido valida JWT
  - Si inválido o expirado → 401 Unauthorized
  - Si usuario inactivo → 401 Unauthorized

AUTORIZACIÓN:

✅ Control de roles:
  - CONSULTA: Lectura de documentos
  - ARCHIVO: CRUD documentos, validación
  - SUPERADMIN: Gestión usuarios + acceso total

✅ Dependencias en FastAPI:
  - require_role("SUPERADMIN"): Validar rol específico
  - get_current_user(): Validar JWT + usuario activo

VALIDACIÓN DE DATOS:

✅ Pydantic BaseModel:
  - Tipos tipados (int, str, bool, datetime)
  - Longitud mínima/máxima
  - Valores permitidos (Literal)
  - Conversiones automáticas

CONTRASEÑAS:

✅ Hash con bcrypt:
  - Mínimo 8 caracteres
  - Rounds automáticos (12 por defecto)
  - Verificación con verify_password()

✅ Nunca retornar:
  - password_hash en respuestas
  - Contraseña en logs
  - Credenciales en URLs

BASE DE DATOS:

✅ SQL Parametrizado:
  - Previene SQL injection
  - cur.execute(query, params)
  - Parámetros con %s

✅ Transacciones:
  - Auto-commit en éxito
  - Auto-rollback en error
  - Contexto manager: with get_db_cursor()
